多线程

join和detach作用及区别

join()等待线程结束，同步方式
含义：主线程等待子线程执行完毕后再继续执行
特点：

  • 是否阻塞：主线程会等待
  • 是否安全：安全，资源会正确释放
  • 声明周期：由主线程管理
  • 使用场景：主线程要依赖子线程的结果时（如计算、加载等）
    detach()线程独立运行，异步方式
    含义：线程脱离控制，变为后台线程（守护线程）。执行结束后，系统会自动回收资源，主线程不能再与之交互。
    特点：
  • 是否阻塞：不阻塞，子线程独立运行
  • 是否安全：可能线程未完成，程序就退出了
  • 使用场景：后台任务（日志、心跳、定时等）

模版类问题

结论：C++模版类的成员函数通常需要在同一个文件中实现（即头文件.h或.hpp文件内）.
现象：若类模版成员函数声明在.h文件，实现在.cpp文件，会报错undefined reference to MyTemplate<int>::print()。
原因：模版在编译时不会立即生成代码，它只是一个“代码生成规则”。只有实例化时，编译器才会根据模版参数生成真正的代码。

回调函数

定义：一段可被作为参数传递给其他函数的代码，并且在适当的时候（由接收该函数的函数确定）被调用。类似于你给别人你的电话号码，然后让他们在需要的时候打给你。

类中禁止拷贝

禁止拷贝原因：类SDLdisplay可能管理着窗口句柄，图形资源，其他不可复制的系统资源。如果可以拷贝，会导致以下问题：

  • 双重释放：两个对象尝试释放同一个资源
  • 资源泄漏：资源丢失无法释放
  • 未定义行为：访问已释放资源

sizeof和stelen区别

sizeof为编译期运算符，用来获取变量或类型所占的字节数。
特点：

  • 在编译期确定大小（不随内容改变）
  • 获取的是对象所占内存大小
  • 用于数组、结构体、基本类型的字节数计算
    strlen用于计算C风格字符串长度
  • 只适用于C风格字符串（char且以\0结尾）
  • 返回长度=字符个数，不包含\0
  • 运行期函数，需要遍历字符直到遇到\0，是O(n)时间复杂度，需要一个字节一个字节的扫描
  • 对非\0结尾的数组使用strlen会导致未定义行为（UB）

头文件的作用

  • libavformat/avformat.h:封装多媒体封装格式（容器）的读写，如mp4、avi或摄像头流
  • libavcodec/avcodec.h:音视频编解码库，用于解码或编码视频帧。
  • libavdevice/avdevice.h:设备输入输出库，用于访问摄像头、麦克风等。
  • libswscale/swscale.h:图像像素格式和分辨率转换库。
  • libavutil/imgutils.h:提供图像缓冲区分配、计算所需空间等工具函数。

AVFrame中常用成员说明

通用数据指针

uint8_t *data[AV_NUM_DATA_POINTERS]
说明：这是一个指针数组，用于存放指向实际平面数据的指针。
对于视频：通常用于存储YUV格式数据。例如：data[0]->Y分量（亮度），data[1]->U分量（色度），data[2]->V分量（色度）。
对于音频：用于存储PCM数据。
AV_NUM_DATA_POINTERS 通常为8

行宽和步长

int linesize[AV_NUM_DATA_POINTERS]
说明：与data数组对应，表示每个平面中一行像素或一个声道的音频数据的字节大小。
对于视频：linesize[0]表示Y平面一行的字节数，linesize[1]表示U平面一行的字节数，linesize[2]表示V平面一行的字节数。由于内存对齐的要求，这个值可能大于图像宽度。例如，一个宽度为1280的YUV420P图像，其linesize[0]可能是1280，也可能是1284或1312等对齐后的值。因此在计算像素位置时，必须使用linesize[0] 而不是图像的宽度。

格式与描述

int format
说明：描述帧的格式。
对于视频：其值是AVPixelFormat枚举（如AV_PIX_FMT_YUV420P），定义了像素的存储格式（色彩空间、平面/打包等）。
对于音频：其值是AVSampleForamt枚举（如AV_SAMPLE_FMT_FLIP），定义了采样数据格式（整数/浮点，平面/交错等）。
AVRational sample_aspect_ratio
说明：像素宽高比，描述了单个像素的宽高比，用于正确显示图像而不变形。例如，标准定义的宽高比为16:9。
int64_t pts
说明：显示时间戳 。表示该帧在什么时间应该被显示出来，单位是AVStream->time_base。这是处理音频同步的关键。
注意：从解码器中出来的帧，其PTS可能不是连续的，或者需要根据AVStream->time_base换算。
int coded_picture_number： 在编码序中的位序号。
int display_picture_number： 在显示序中的帧序号。

视频特定成员

int width, height
说明：图像的宽度和高度，以像素为单位。
int key_frame
说明：标识这一帧是否为关键帧（I帧）。0表示不是，1表示是。
enum AVPictureType pict_type
说明：帧的类型

  • AV_PICTURE_TYPE_I：I帧
  • AV_PICTURE_TYPE_P：P帧
  • AV_PICTURE_TYPE_B：B帧
  • AV_PICTURE_TYPE_S：S帧
  • AV_PICTURE_TYPE_SI：SI帧
  • AV_PICTURE_TYPE_SP：SP帧

AVRational repeat_pict
说明：指示在解码时，某些格式（如MJPEG）中B帧需要延时多久再显示。用于更精确的计算PTS。
int flags
说明：在FFmpeg中，每个AVPaket都有一个int flags字段，用于标识数据包的一些特殊属性或状态标志。这些标志是按位掩码定义的，也就是说可以同时存在多个标志 （用按位或｜组合）。

常用可选值

宏名 含义
AV_PKT_FLAG_KEY 表示该包是关键帧（I帧），可以单独解码
AV_PKT_FLAG_CORRUPT 表示该包数据损坏（例如网络丢包导致）
AV_PKT_FLAG_DISCARD 表示该包可被丢弃（非必要数据）
AV_PKT_FLAG_TRUSTED 表示该包来自可信源
AV_PKT_FLAG_DISPOSABLE 表示该帧可被忽略（非关键帧且仅临时辅助使用）
AV_PKT_FLAG_SKIP 表示解码器可跳过此包（很少见）
AV_PKT_FLAG_DECRYPTED 表示包已被解密（用于加密流）
AV_PKT_FLAG_FRAME 表示该包对应一个完整帧（通常在封装层使用）

缓冲区和引用计数

AVFrame通常不直接分配数据内存，而是引用底层的AVBuffer。

  • AVBufferRef *buf[AV_NUM_DATA_POINTERS]
    说明：与data数组对应的缓冲区引用。如果buff[i]不为NULL，则表示data[i]指向的内存是由buff[i]管理的。这是FFmpeg实现引用计数的核心。
  • int ref_count
    说明：是AVFrame的引用计数。当引用计数减为0时，帧会被自动释放。不要直接修改该值，应使用av_frame_ref和av_frame_unref等API来管理。

数据格式

平铺格式（Planar）

特点：

  • 每个分量在内存中完全分开
  • 处理某个分量时很方便

优势：

  • 处理效率高：如果只需要灰度图像（亮度Y），直接操作data[0]即可
  • 压缩友好：色度分量可以单独进行下采样（例如nv12nv21）
  • 并行处理：不同平面可以同时处理

实用场景：

  • 视频编解码（H.264等)
  • 图像处理算法
  • 色彩空间转换

打包格式

特点：

  • 每个像素的各个分量紧挨排列
  • 适合直接展示，处理单个分量不方便

优势：

  • 显示友好：直接对应显示硬件需求
  • 内部局部性：一个像素的所有分量都在一起
  • 传输效率：适合直接发送到显卡

1、创建socket套字节

头文件：
#include<sys/types.h>
#include<sys/socket.h>
int socket(int domain, int type, int protocol)
功能：创建一个具有网络属性的文件描述符。
参数：
domain：协议族

  • AF_UNIX、AF_LOCAL 本地连接
  • AF_INET IPv4
  • AF_INET6 IPv6

type： 

  •  SOCK_STREAM：流式套字节，面向连接的字节流TCP。保证顺序且可靠（重传）

  •  SOCK_DGRAM：数据报套字节，面向无连接的数据报UDP。每次`sendto/recvfrom`保持报文边界

  •  SOCK_RAW： 原始套字节，直接访问网络层或链路层（需要特权）。用于实现自定义协议或抓包。

  •  SOCK_SEQPACKET：有连接的顺序报文（介于STREAM和DGRAM之间，保留报文边界且有连接性）。

如何选：要做TCP用SOCK_STREAM，要做UDP用SOCK_DGRAM。要做自定义协议或抓包用SOCK_RAW（需要root或管理员权限）。
protocol：指定具体的传输协议（协议号）。多数时候该参数设置为0表示使用domain+type对应的默认协议（例如AF_INET+SOCK_STREAM默认是TCP）。

  • 0:让内核挑选合适的协议。
  • IPPROTO_TCP：TCP（通常用SOCK_STREAM明确指定）
  • IPPROTO_UDP：UDP（通常用SOCK_DGRAM明确指定）
  • IPPROTO_IP / IPPROTO_RAW：用于原始套字节

返回值：
成功：返回一个非负的文件描述符
失败：返回-1，并设置错误码。

3、监听

int listen(int sockfd, int backlog);
功能：将一个已绑定的套字节设置为被动监听状态，告诉操作系统：我准备好接收数据了。通常用于TCP连接，UDP不需要监听。
参数：
sockfd—套字节文件描述符

  • 必须是TCP套字节（SOCK_STREAM）
  • 必须已绑定到本地IP+端口，如果没绑定，调用listen会失效

backlog—半连接队列长度
backlog决定操作系统为这个监听socket分配的等待连接队列长度。
队列概念
TCP三次握手过程：

  1. 客户端发送SYN
  2. 服务端收到，发送SYN-ACK（进入到半连接队列）
  3. 客户端发送ACK->建立连接，进入已连接队列
    backlog定义的就是半连接队列（尚未完成三次握手的连接）长度上限。
  • 如果半连接队列满了，新的客户端尝试连接会被拒绝或等待。
  • 一般Linux内核会把backlog取系统允许最大值

返回值：
成功：返回0
失败：返回-1，并设置错误码

4、等待客户端连接（等待意味着“阻塞”accept）

int accept(int sockfd, struct sockaddr *addr, socklen_t *addrlen);
功能：从全连接队列中取出一个已完成的连接，返回一个新的描述符（socket）用于和该客户端通信。原来的sockfd仍然保持坚挺状态，可以继续接收更多连接。
参数详解：

  • sockfd：已经监听的套字节（必须是SOCK_STREAM类型）
  • strcut sockaddr *addr:(输出参数，可选)指向内存，用于存储建立连接的远端客户端地址，如果不需要存储则传入NULL
  • addrlen ：结构体长度。调用前应设置为sizeof(*addr)；调用返回时会被修改为实际写入addr的字节长度。

返回值：
成功：返回非负的新文件描述符。
失败：返回-1，并设置错误码

accept与TCP三次握手/队列的关系

  • 当客户端发起连接并完成三次握手后，内核会把该连接放入到已完成连接队列，accept会从该队列中取出并返回一个新的fd。
  • listen(backlog)控制内核在挂起状态允许的并发连接数
  • 如果半连接/已连接队列已满，新的连接可能会被拒绝或丢弃（客户端会看到连接失败或超时）

信号槽机制

信号槽机制特点：

  • 类型安全：参数类型必须匹配
  • 松耦合：发送者不知道接收者
  • 多对多：一个信号和连接多个槽，一个槽可以接收多个信号

信号（Signals）

  • 只有声明，没有实现
  • 返回值必须是void
  • 可以是带参数，也可以是不带参数
  • 在signals:区域进行声明

连接（Connect）
连接语法

连接类型：
自动连接（AutoConnection）–默认

行为：如果发送者和接收者在同一线程，使用DirectConnection；否则使用QueuedConnection
使用场景：大多数情况下默认选择
直连（DirectConnection）

行为：信号发射时立即调用槽函数，就像函数调用一样
线程安全：必须在同一线程
使用场景：性能要求高，且在同一线程
队列连接（QueuedConnection）-跨线程

行为：将槽函数调用放入接收者线程的事件队列，异步执行
线程安全：可以跨线程，参数会被拷贝
使用场景：跨线程通信
阻塞队列连接（BlockingQueuedConnection）

行为：类似于QueuedConnection，但会阻塞发送者线程直到槽函数执行完成
线程安全：跨线程，但发送者和接收者必须在不同线程
